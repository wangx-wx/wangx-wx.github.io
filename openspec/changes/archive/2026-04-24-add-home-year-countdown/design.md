## Context

当前首页没有展示年度时间进度的信息。该需求是一个前端单页内增强，数据完全由客户端时间推导，无需后端接口。项目使用 Astro 构建页面，因此倒计时组件应采用客户端激活（hydration）方式，避免在服务端渲染阶段引入“构建时刻”与“用户访问时刻”不一致问题。

## Goals / Non-Goals

**Goals:**
- 在首页新增一个可复用的年度倒计时展示模块。
- 展示本年度已过比例、已过去时长、剩余时长，并提供天/时/分/秒实时倒计时。
- 组件在客户端每秒更新，保证可见数据连续、准确。
- 在跨年时自动切换到新年度并继续倒计时。

**Non-Goals:**
- 不引入服务端时间同步或 NTP 校时能力。
- 不提供历史年份查询或自定义目标日期。
- 不改动站点其他页面的信息架构。

## Decisions

1. 使用独立前端组件承载倒计时逻辑（例如 `YearCountdown`）。
- Rationale: 将时间计算与首页布局解耦，便于后续复用、测试和样式调整。
- Alternative: 直接将逻辑写入首页模板；实现更快但可维护性差。

2. 时间计算全部基于用户本地 `Date`。
- Rationale: 需求是“今年”倒计时，语义应与访问者所在时区一致，无需网络请求。
- Alternative: 使用固定时区（如 UTC+8）统一展示；会与部分用户本地“今年剩余时间”认知不一致。

3. 使用 `setInterval` 按 1 秒更新，并在组件卸载时清理计时器。
- Rationale: 秒级精度已满足需求，实现简单稳定。
- Alternative: `requestAnimationFrame` 可更平滑，但对该文本场景属于过度设计。

4. 进度值按“当前时间 - 年初 / 全年总毫秒”计算，并限制在 `[0,1]`。
- Rationale: 防止边界时刻出现浮点误差导致的越界显示。
- Alternative: 由剩余时间反推进度；效果等价，但边界处理更分散。

## Risks / Trade-offs

- [用户手动修改系统时间导致跳变] → 接受该行为；组件在下一次 tick 自动按新时间重算。
- [后台标签页定时器节流导致秒数短暂不连续] → 每次 tick 全量重算而非递减缓存值，恢复前台后可自动纠正。
- [跨年瞬间出现 0 或负值闪烁] → 统一通过 `Math.max(0, diffMs)` 与边界钳制处理，并在 `diffMs <= 0` 时重建新年度边界。

## Migration Plan

- 仅前端代码变更，直接随常规站点发布。
- 回滚方式：移除首页组件引用或回退对应提交，不涉及数据迁移。
